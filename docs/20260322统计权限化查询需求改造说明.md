# 统计查询权限化需求改造说明（2026-03-22）

## 1. 背景与目标
当前统计模块以实时聚合为主，按请求对业务表进行 `SUM/GROUP BY` 计算，未对“不同账号看到不同信息”进行细粒度控制。

本次需求目标：
1. 保持现有统计能力（累计、季度、年度、项目报告）可用。
2. 新增“按账号权限返回不同统计信息”的查询能力。
3. 明确哪些信息实时计算，哪些可选持久化缓存。

## 2. 当前现状（已确认）
1. 统计结果未单独存储到数据库，仅实时计算返回。
2. 原始数据存储在 `market_data`、`engineering_data`、`finance_data`。
3. 统计接口当前仅校验登录态与激活状态，未做部门级可见性控制。

## 3. 需求范围

### 3.1 必做范围
1. 新增统一统计查询接口（支持模块、时间粒度、时间范围、项目过滤）。
2. 增加权限控制策略：依据账号部门返回不同模块和字段。
3. 增加项目访问范围控制：用户仅可查询有权限的项目。
4. 增加审计信息：记录查询人、查询参数、查询时间（可先做日志级）。

### 3.2 可选增强
1. 高频统计缓存（短时缓存）。
2. 统计快照表/物化视图（用于大数据量场景）。
3. 导出功能（按权限导出字段）。

## 4. 需要修改的文件与模块

## 4.1 后端 Schema 层
1. `backend/app/schemas/statistics.py`
- 保留现有汇总模型。
- 可新增统一查询请求/响应模型（或拆分到 `statistics_query.py`）：
  - 请求：`module`, `granularity`, `start_year`, `end_year`, `project_id`, `include_detail`
  - 响应：`data`, `totals`, `visible_fields`, `scope`

## 4.2 后端 Service 层
1. `backend/app/services/statistics_service.py`
- 新增可复用方法：
  - `build_access_scope(user, module)`
  - `apply_project_scope(query, scope)`
  - `mask_fields_by_scope(result, scope)`
- 现有 `get_market_summary/get_engineering_summary/get_finance_summary/get_project_report` 增加权限参数并执行范围过滤。

2. 可选新增文件：`backend/app/services/statistics_query_service.py`
- 承载统一查询逻辑，避免在单个 service 文件中堆叠过多条件分支。

## 4.3 后端 Router 层
1. `backend/app/routers/statistics.py`
- 新增统一查询接口，例如：`POST /api/statistics/query`。
- 现有接口补充权限控制：
  - 非 admin 用户按部门限制模块可见性。
  - `project/{project_id}/report` 增加项目访问校验。

## 4.4 权限依赖与用户模型
1. `backend/app/utils/dependencies.py`
- 增加可复用统计权限依赖或策略函数（如 `require_statistics_scope`）。

2. `backend/app/models/user.py`
- 若仅按部门控制，可不改表结构。
- 若需要项目级授权，新增成员关系表（见数据库部分）。

## 4.5 数据库层（仅保留方案 A+）

### 方案 A+：部门控制 + 管理员临时扩权（本次推荐）
- 默认仍按 `users.department` 控制可见范围（保持方案A简单性）。
- 增加“临时扩权”能力：管理员可给指定用户开通跨部门查询，且带失效时间。
- 临时扩权到期后自动失效，无需手工回收。

## 4.6 前端层
1. `frontend/src/services/api.ts`
- 新增统一统计查询 API 方法（如 `statisticsApi.query()`）。

2. `frontend/src/types/index.ts`
- 增加统一查询请求/响应类型。
- 增加权限相关可见字段描述类型。

3. `frontend/src/pages/Statistics.tsx`
- 新增查询表单（模块、时间范围、项目）。
- 根据后端返回的 `visible_fields` 动态渲染图表/表格列。

## 5. 权限设计建议（第一版）
1. `admin`：全部模块 + 全部字段 + 全部项目。
2. `market`：市场完整；工程/财务仅汇总级字段。
3. `engineering`：工程完整；市场/财务仅汇总级字段。
4. `finance`：财务完整；市场/工程仅汇总级字段。

## 5.1 临时扩权设计（管理员）
1. 目标：让非管理员账号在限定时间内查看额外模块数据（跨部门查看）。
2. 触发方式：仅管理员可发起授权（可通过后台管理页面或管理接口）。
3. 约束：
- 必须设置 `expires_at`（禁止永久临时授权）。
- 必须记录 `granted_by`、`reason`（用于审计）。
- 授权可提前撤销。

## 5.2 临时扩权数据结构（推荐最小实现）
在不引入项目成员体系的前提下，建议新增一张轻量授权表：

`statistics_access_grants`
1. `id` UUID 主键
2. `user_id` 被授权用户
3. `module` 授权模块（market/engineering/finance）
4. `scope` 授权范围（all_projects 或单项目）
5. `project_id` 可选（当 scope=single_project 时必填）
6. `expires_at` 授权到期时间
7. `granted_by` 管理员用户ID
8. `reason` 授权原因
9. `is_active` 是否有效
10. `created_at` 创建时间

> 如需严格坚持“无新增表”，可使用缓存存储临时授权，但在多实例部署、审计追踪、故障恢复方面风险较高，不建议用于生产。

## 5.3 权限判定顺序（服务端）
1. 先按部门基础权限判定。
2. 若基础权限不足，检查临时授权是否存在且未过期。
3. 若存在有效临时授权，按授权范围返回数据。
4. 否则返回受限结果（403或空数据，按产品约定）。

## 5.4 需要新增/修改的接口
1. 管理员授权接口：
- `POST /api/statistics/access-grants`（创建临时授权）
- `DELETE /api/statistics/access-grants/{grant_id}`（提前撤销）
- `GET /api/statistics/access-grants`（查询授权记录）
2. 统计查询接口：
- 统一查询入口在返回前调用 `resolve_access_scope(user, module, project_id)`。

## 5.5 需要修改的代码位置（增量）
1. `backend/app/services/statistics_service.py`
- 增加临时授权判定逻辑。
2. `backend/app/routers/statistics.py`
- 增加授权管理路由（管理员可用）。
3. `backend/app/utils/dependencies.py`
- 增加管理员校验复用（已存在 `require_admin`，可直接复用）。
4. 数据库迁移（alembic）
- 新增 `statistics_access_grants` 表。

## 5.6 验收补充（临时扩权）
1. 被授权用户在授权期内可访问额外模块数据。
2. 到期后自动恢复到部门基础权限。
3. 管理员可提前撤销并即时生效。
4. 所有授权动作与查询动作可审计。

> 注：字段级控制建议由后端输出最终可见数据，前端仅做展示，不在前端做敏感字段裁剪。

## 6. 累计口径约定（避免歧义）
1. 月累计：同一自然年内按月份累加。
2. 季度累计：按季度聚合（Q1-Q4）。
3. 年累计：按年度聚合。
4. 比率项：
- 审批率 = `total_approval / total_output * 100`
- 毛利率 = `(total_revenue - total_cost) / total_revenue * 100`

## 7. 实施顺序（建议）
1. 先落地方案 A+（部门基础权限 + 管理员临时扩权 + 统一查询接口）。
2. 稳定后按性能情况评估是否增加统计缓存/快照优化。

## 8. 验收标准
1. 不同部门账号查询同一条件时，返回内容存在可预期差异。
2. 无权限项目不可查询，返回 403/空结果（按产品约定）。
3. 累计、季度、年度口径一致且可复核。
4. 原有统计接口兼容不破坏。
5. 日志可追踪查询人、查询时间、查询条件。

## 9. 风险与注意事项
1. 若只做前端隐藏字段而后端不裁剪，存在数据泄漏风险。
2. 动态字段权限会影响图表配置，需统一前端渲染策略。
3. 大数据量实时聚合可能导致慢查询，应预留索引与缓存方案。
